Ride

Summary

RIDE was a project done in the IAT267 class where teams were tasked with creating an interactive system combining physical sensors and coding. Much of the work I did was in the sensor code which required restructuring mid project. Above is a demo of the finished product.



Process

When designing this project, the concept I had in mind was “an intuitive game controller.” The idea that I came up with was a game where the player controlled a vehicle via pedaling on a stationary bike. We were able to achieve this by using Arduino to read physical sensors and transferring the data to Processing where the game ran. After deciding on what we wanted to create, we brainstormed ideas for the form of the project, and how users will interact with it. The thing we decided that we definitely wanted was that the pedalling speed will control the character's speed on screen, so we built around this idea.



Challenges

The first task was figuring out how the we would control the speed of the in-game character with the speed of the pedaling. Initially my idea was to use a light sensor in conjunction with an LED attached to the wheel of the bike to calculate the frequency at which the wheel rotated. However, the readings from the LED were unreliable because the light sensor picked up the ambient light. My solution to this was to attach a magnet to the wheel of the bike and a magnet sensor to the inner frame which made readings much more reliable. The idea behind this was that the sensor would receive a positive reading every time the wheel made a full rotation and the magnet passes the sensor. Then, the program would count the number of times the magnet passed within a set interval. The problem with this method was that the speed of the game character would only update after each interval which made changes in speed feel unresponsive. To circumvent this problem, I tried a different approach to adjusting the character’s speed. Instead of calculating the number of rotations in a given interval, I changed the system to add a decaying acceleration to the character with each positive reading from the sensor. Essentially, this adds “gas” to a tank which changes the speed depending on how quickly the tank filled and emptied. This allowed for a smoother transition between speeds which reduced choppiness. From working on this project, I learned that sometimes it is more effective to explore other methods if the initial idea doesn’t work out. Instead of trying to make the rotation frequency code work, I took a step back to analyze and restructure how it operated on the fundamental level.
